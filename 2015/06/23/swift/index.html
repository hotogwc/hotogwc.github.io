WWDC 2015笔记：Swift 2

发表于 2015-06-23

各种事，时隔一周才开始写这篇，这个不是某一个 session 的笔记，是几个关于 Swift 的 session 的总结。把我自己感兴趣的点都研究了一下，大家如果想看更多其他的细节，可以自行去看视频。同时，由于有了吊炸天的支持 markdown 的 playground，所以其实类似这种传统博客写成 playground 会更帅，大家也能自己在 playground 上改代码看效果，所以这次再配一个 playground 版本，有 Xcode 7 的同学可以下载交流. 关于新 playground ，大家可以看这个session。playground 链接在这：

Enum

Enum 本身就是 Swift 最强悍的特性，这次在 Swift 2 也得到了很多的增强，首先就是打印的结果有了优化，比如下边的代码。

enum Animals {
  case Dog, Cat, Troll, Dragon
}

let a = Animals.Dragon

之前的 Swift，打印的结果是(Enum Value),这个结果对于我们的帮助非常有限，改进之后，现在的输出是Animals.Dragon 。是不是更清晰了呢？

然后是关联值的变化，在之前的版本，我们可以在 Enum 中添加关联值 （associated value），可以使我们通过 Enum 传值，但是值的类型是固定的，现在在 Swift 2 里，我们可以在 Enum 的关联值里传不同的类型了，比如：

enum Either<T1, T2> {
  case First(T1)
  case Second(T2)
}

这么写之前是会报错的，现在就不会，这是非常实用的改进，比如我们有一个网络请求，在失败的时候可以直接传来需要展示给用户看的信息，成功的时候传进来数据，非常好用。

Guard

这个好玩了，大家在第一次看的时候肯定会想，guard 并没有什么卵用啊，看不出跟 if 有什么区别啊。这里我要把 guard 这个东西深入展开一下，让大家看看它好在哪里。先来看官方的解释

Like an if statement, guard executes statements based on a Boolean value of an expression. Unlike an if statement, guard statements only run if the conditions are not met. You can think of guard more like an Assert, but rather than crashing, you can gracefully exit.

看了还是有点晕吧，咱们来用用看。

func fooCheck(x: Int?) {
    if x == nil || x <= 0 {
        // 未达成条件
        return
    }

    //达成条件
    x!.description
}

虽然是 Swift 代码，其实还是 OC 的套路，这么写的缺点是你是在检查你不希望达到的条件，如果达成的话，你还需要写 x! 给变量强制解包。
在 Swift 1.2 里，我们又可以这么写：

func fooBinding(x: Int?) {
    if let x = x where x > 0 {
        // 达成条件
        x.description
    }

    // 未达成
}

看上去好多了，但是这样还有个小问题： 你想要执行的代码在所有的检查条件的内部，如果检查的条件很多，嵌套了很多层，代码就会变得很乱，可读性非常差，如果代码最后没执行，你也不知道是这些嵌套里的哪个条件没有达成导致的。（如果你用 Swift 解析 JSON 而且不用 SwiftyJSON 这个库的时候你就知道这有多蛋疼了。）

我们希望的是可以有非常清晰的条件检查而且最好顺便帮我们把可选变量都解包，if let大家应该写够了吧。

咱们的 guard 出场啦：

func fooGuard(x: Int?) {
    guard let x = x where x > 0 else {
        // 条件未达成，跳出。
        return
    }

    // 达成
    x.description
}

3个好处：

  • 你检查的是你希望达成的条件，类似一个断言。
  • 如果条件达成，可选变量就解包，可以拿来用了。
  • 在做你想做的事儿以前就检查了条件，如果没达成就提前退出了。如果条件很多，很容易就会发现哪个条件没有达成。

貌似例子写的简单了点，那么大家看底下这个解析JSON的代码片段，配合可以传多类型的 Enum，让大家看看 Swift 2 的威力。

func process(json: AnyObject) -> Either<Person,String> {
    guard let name = json["name"] as? String,
          let year = json["year"] as? Int else {
        return .Second(“bad input”)
    }
    let person = processPerson(name, year)
    return .First(person)
}

Defer

又一个好东西，Defer 后边跟一个闭包，这个闭包的会在当前代码段退出之前执行。是回调闭包，回调代理方法，清理对象最完美的地方。而且这闭包写在代码段的那个位置都可以。看下边的例子。

func processSale(json: AnyObject) throws {
  delegate?.didBeginReadingSale()
  guard let buyerJSON = json["buyer"] as? NSDictionary {
    throw DataError.MissingBuyer
  }
  let buyer = try processPerson(buyerJSON)
  guard let price = json["price"] as? Int {
    throw DataError.MissingPrice
}
  return Sale(buyer, price)
}

我们在开始这个方法的时候回调了 didBeginReadingSale() 代理方法，这时我们希望在这个方法退出的时候调用 didEndReadingSale() 代理方法，问题是在哪调用呢？ 每一个 guard 都有判断失败抛出异常退出方法的可能性。安全起见，我们可以这么写：

func processSale(json: AnyObject) throws {
  delegate?.didBeginReadingSale()
  guard let buyerJSON = json["buyer"] as? NSDictionary {
    delegate?.didEndReadingSale()
    throw DataError.MissingBuyer
  }
  let buyer = try processPerson(buyerJSON)
  guard let price = json["price"] as? Int {
    delegate?.didEndReadingSale()
    throw DataError.MissingPrice
}
  delegate?.didEndReadingSale()
  return Sale(buyer, price)
}

怎么样？丑的要死吧，刚才我们说了 defer 会在一个代码段退出的时候执行，而且书写的顺序不限，那么我们就可以在调用 didBeginReadingSale() 之后马上用 defer 再调用完成的代理方法 didEndReadingSale() 。 这样不管 processSale 在哪退出，都会在退出前调用 didEndReadingSale() 。

func processSale(json: AnyObject) throws {
  delegate?.didBeginReadingSale()
  defer { delegate?.didEndReadingSale() }
  guard let buyerJSON = json["buyer"] as? NSDictionary {
    throw DataError.MissingBuyer
  }
  let buyer = try processPerson(buyerJSON)
  guard let price = json["price"] as? Int {
    throw DataError.MissingPrice
  }
  return Sale(buyer, price)
}

酸爽吗？
